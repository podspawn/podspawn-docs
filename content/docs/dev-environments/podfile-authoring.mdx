---
title: Writing a Podfile for Your Repo
description: Guide for repo owners who want contributors to have a one-command setup experience.
---

A Podfile committed to your repo means every contributor, CI runner, and AI agent gets the exact same development environment. No "works on my machine," no 30-minute setup docs.

## Where to put it

<Files>
  <Folder name="your-project" defaultOpen>
    <File name="podfile.yaml" />

      <File name="podfile.yaml" />

Place `podfile.yaml` at the root of your repository. Alternatively, use `.podspawn/podfile.yaml` if you prefer keeping it in a dotdir. Podspawn checks both locations, with the root file taking priority.

## Minimal examples

<Tabs items={['Go', 'Node.js', 'Python']}>
  <Tab value="Go">
```yaml
base: ubuntu:24.04
packages: [go@1.24, make]
on_create: go mod download
```
  </Tab>
  <Tab value="Node.js">
```yaml
base: ubuntu:24.04
packages: [nodejs@22]
on_create: npm install
```
  </Tab>
  <Tab value="Python">
```yaml
base: ubuntu:24.04
packages: [python@3.13]
on_create: |
  pip install uv
  uv sync
```
  </Tab>
</Tabs>

This gives contributors the language toolchain and pre-downloaded dependencies. That's often enough.

## A real-world example

```yaml
extends: ubuntu-dev
packages:
  - go@1.24
  - make
  - protobuf-compiler

env:
  DATABASE_URL: "postgres://postgres:devpass@postgres:5432/myapp"

services:
  - name: postgres
    image: postgres:16
    ports: [5432]
    env:
      POSTGRES_PASSWORD: devpass
      POSTGRES_DB: myapp

ports:
  expose: [8080, 9090]
  strategy: auto

on_create: |
  go mod download
  go install google.golang.org/protobuf/cmd/protoc-gen-go@latest
  make generate

on_start: |
  echo "Ready. Run 'make dev' to start the server."
```

## Best practices

**Pin package versions.** `go@1.24` is better than just `go`. Contributors should all be on the same version.

**Keep it minimal.** Only include what's needed to build and test. Personal preferences (editor configs, shell themes) belong in dotfiles, not the Podfile.

**Use `on_create` for heavy setup.** Dependency installation, code generation, database migrations -- anything slow goes in `on_create` because it only runs once when the container is first created.

**Use `on_start` for lightweight tasks.** Welcome messages, environment checks, starting background processes.

**Use `extends` for shared bases.** If your org has multiple repos with similar tooling, create a base Podfile and extend it rather than duplicating setup across repos.

**Test with `podspawn dev`.** Before committing, run `podspawn dev -- make test` to verify the Podfile produces a working environment.

## What not to include

<Callout type="warn">
Never put API keys, passwords, or credentials in a Podfile. It gets committed to git. Use environment variables injected at runtime instead.
</Callout>

<Accordions>
  <Accordion title="Large datasets">
    Don't bundle data files in the Podfile. Mount them as volumes or download them in `on_create`. If the data is shared across projects, use a named volume so it persists across sessions.

    ```yaml
    services:
      - name: data
        image: busybox
        volumes:
          - shared-data:/data
    ```
  </Accordion>
  <Accordion title="IDE-specific config">
    VS Code extensions, vim plugins, Emacs packages, shell themes -- these belong in your personal [dotfiles](/docs/dev-environments/reference#dotfiles), not the Podfile. Each contributor can bring their own editor setup via the `dotfiles` field.

    ```yaml
    dotfiles:
      repo: github.com/youruser/dotfiles
      install: ./install.sh
    ```
  </Accordion>
  <Accordion title="Secrets and credentials">
    API keys, tokens, and database passwords for real services should never appear in a Podfile. Use environment variables injected at runtime, or a secrets manager. Dev-only passwords for local services (like `POSTGRES_PASSWORD: devpass`) are fine since they only exist inside the container.
  </Accordion>
</Accordions>
