Podspawn is a single binary for dev container workspaces. It works two ways: locally via Docker exec (no SSH, no root) and remotely via native sshd (zero client install). One tool, two modes, zero friction.

```bash
# Local -- create a container and drop in
podspawn create dev && podspawn shell dev

# Server -- SSH in, land in a container
ssh alice@devbox.company.com
```

## Two modes, same tool

**Local mode** talks to Docker directly. No SSH, no sshd, no root, no key management. If `docker ps` works, podspawn works. Named containers persist across shells, or use `podspawn run scratch` for one-shot throwaways.

**Server mode** hooks into your host's native sshd via `AuthorizedKeysCommand`, turning two lines of sshd_config into a full container workspace platform. Users SSH in with any client and land in isolated containers. Every SSH feature works -- port forwarding, SCP, SFTP, VS Code Remote -- because OpenSSH handles the protocol.

Same binary, same Podfile environments, same container lifecycle. The only difference is how you get in.

## What makes it different

- **Podfile environments** -- declarative YAML for packages, companion services, dotfiles, and lifecycle hooks. Define your dev environment once, reproduce it anywhere.
- **Named machines** -- `podspawn create backend`, `podspawn shell backend`. Containers have names, not hex IDs.
- **Security by default** -- cap-drop ALL, no-new-privileges, PID limits, per-user network isolation, optional gVisor.
- **AI agent ready** -- disposable environments for agents, persistent workspaces for teams. Create, run tests, destroy. No cleanup.
- **Cross-platform** -- macOS, Linux, Windows. Works with OrbStack, Docker Desktop, Colima, or bare Docker.
- **Persistent or ephemeral** -- per-user, per-project, admin's choice.
- **Single binary, open source** -- no daemon in local mode, no custom SSH server in server mode. AGPL-3.0.

## Why not...

**Docker directly?** You can, but you'll reinvent named machines, Podfile-driven environments, companion services, and lifecycle management. Podspawn is the porcelain over Docker's plumbing.

**OrbStack?** Excellent on macOS, but macOS-only and closed source. Podspawn is cross-platform, Podfile-driven, has server mode for teams, and is open source.

**DevPod?** Similar goals, heavier approach. DevPod runs a daemon, requires a provider system, and doesn't have transparent SSH server mode. Podspawn is a single binary with no moving parts.

**Dev Containers / devcontainer CLI?** Tightly coupled to VS Code. Podspawn works with any editor, any terminal, any SSH client.

## Quick start

### Local (30 seconds)

```bash
curl -sSfL https://podspawn.dev/up | bash   # choose "Local"
podspawn create dev
podspawn shell dev
```

### Server (2 minutes)

```bash
curl -sSfL https://podspawn.dev/up | bash   # choose "Server"
# Configures sshd, registers your keys

# From any machine:
ssh alice@yourserver.com
```

## Architecture at a glance

Podspawn is a single binary that serves multiple roles:

**Local** -- `podspawn create` and `podspawn shell` call Docker directly. No daemon, no background process. The container stays alive between shells; `podspawn stop` removes it.

**Server side** -- sshd invokes `podspawn auth-keys` to look up SSH keys. If the user is a container user, it returns keys with a `command=` directive that forces `podspawn spawn`, which creates or reattaches a Docker container and pipes I/O.

**Client side** -- the same binary provides `podspawn connect` as a `ProxyCommand` in `~/.ssh/config`, creating a `.pod` namespace. `ssh alice@work.pod` is intercepted before DNS, routed to the right server, and the user lands in their container. The client binary is optional; any SSH client works without it.
