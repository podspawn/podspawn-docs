---
title: podspawn shell
description: Attach to a local machine with an interactive shell
---

Attaches to a running podspawn machine with an interactive shell. This is the primary way to connect to local machines.

## Usage

```bash
podspawn shell <name>
podspawn shell <user@name>
```

## Arguments

| Argument | Required | Description |
|----------|----------|-------------|
| `name` | Yes | Machine name to attach to |
| `user` | No | Username prefix, same `user@host` format as other commands |

## Examples

```bash
# Attach to a machine
$ podspawn shell dev

# Attach as a specific user
$ podspawn shell alice@backend
```

## How it works

`podspawn shell` uses Docker exec to open a shell inside the container. No SSH is involved, so there is nothing to configure and no port conflicts to worry about.

For local development, this is all you need. SSH is only required for remote access or IDE integration (VS Code Remote, Cursor, JetBrains Gateway).

## Limitations

Because `podspawn shell` uses Docker exec instead of SSH, some features are unavailable:

| Feature | Available? | Workaround |
|---------|-----------|-------------|
| Interactive shell | Yes | -- |
| SFTP / file transfer | No | `docker cp` |
| `scp` | No | `docker cp` |
| Port forwarding (`-L`, `-R`) | No | `docker run -p` or manual `socat` |
| VS Code Remote SSH | No | Use full SSH setup |
| Cursor Remote | No | Use full SSH setup |
| JetBrains Gateway | No | Use full SSH setup |
| SSH agent forwarding | No | Mount `SSH_AUTH_SOCK` manually |

For full IDE and file transfer support, you need a real SSH connection via `podspawn ssh` or direct SSH to the server.
